# 品質基準（note / X 特化）

god-writing 原則に従属する品質評価フレームワーク。
committee_review および執筆後セルフチェックで使用する。

---

## 品質5階層

| Level | 定義 | 判定 |
|-------|------|------|
| 5（卓越） | 読者の期待を超える。「共有したい」が自然に起きる | 理想 |
| **4（優良）** | **ミサキ層が最後まで読み「やってみよう」と思える** | **目標ライン** |
| 3（合格） | 情報として成立するが、技術ブログと同じ書き方でミサキ層への噛み砕きがない | 差し戻し |
| 2（不足） | 読者不在。書き手目線の情報整理になっている | 再設計 |
| 1（不可） | 目的未達成。驚き屋ムーブや事実誤認を含む | 破棄 |

**Level 3 は不合格。** 技術ブログ（Qiita/Zenn等）は品質が高いが読者はエンジニア向け。同じ書き方ではミサキ層（AI初学者）に刺さらない。難しい話を噛み砕いて届けるのがこのアカウントの存在意義。

---

## 5つの評価軸

### 軸1: 目的達成度（Purpose Achievement）

この記事は何のために存在するのか。

| Level | 基準 |
|-------|------|
| 5 | 読者が想定以上の行動を起こす（共有・実践・フォロー） |
| 4 | 目的を明確に達成。読了後に「やってみよう」が残る |
| 3 | 目的をほぼ達成するが、読後の行動が曖昧 |
| 2 | 目的の一部のみ。情報提供で終わっている |
| 1 | 目的が不明確。何のための記事かわからない |

確認項目:
- 目的の階層（L1〜L5）を定義したか
- 全ての段落が目的に貢献しているか
- CTAが目的と整合しているか
- 「情報を並べただけ」になっていないか

### 軸2: 読者適合度（Reader Fit）

ミサキ層（22-25歳、事務・営業・マーケ）が「自分のことだ」と感じるか。

| Level | 基準 |
|-------|------|
| 5 | 読者の言語化できていない悩みまで拾っている |
| 4 | 読者のペインに直接刺さり、理解レベルに合っている |
| 3 | 読者のニーズをほぼ満たすが、一部が専門的すぎる |
| 2 | 読者のニーズとズレている。エンジニア向けの粒度 |
| 1 | 読者を無視している。書き手が書きたいことを書いている |

確認項目:
- エンティティ（読者の心理・状況）を生成したか
- 専門用語は読者が知っている言葉に変換したか
- 「コード書いてください」「ターミナルを開いて」が出ていないか
- 読者の日常業務の文脈で書いているか

### 軸3: 論理構造（Logical Structure）

構造が内容を支えているか。読者が迷わず読めるか。

| Level | 基準 |
|-------|------|
| 5 | 見出しだけで内容が把握でき、本文で深まる |
| 4 | 論理的に整合し、読者の思考フローに沿っている |
| 3 | 概ね論理的だが、情報の並び順に改善余地がある |
| 2 | 論理の飛躍がある。機能一覧の羅列になっている |
| 1 | 構造が破綻。読者が迷子になる |

確認項目:
- 情報の順序は読者の疑問の順序に沿っているか
- 見出しは内容を的確に表しているか（説明的すぎないか）
- 図解の挿入ポイントは「文章だけでは伝わらない箇所」か
- 掛け合いと情報パートが分離されているか

### 軸4: 表現品質（Expression Quality）

「先輩が隣で見せてくれる」温度感があるか。

| Level | 基準 |
|-------|------|
| 5 | 読者の心に残るフレーズがある。声に出して読みたくなる |
| 4 | 自然で読みやすい。書き手の体温が感じられる |
| 3 | 理解可能だが、エンジニア向けの書き方のままで初学者への噛み砕きがない |
| 2 | 不明瞭、冗長、またはトーンが不統一 |
| 1 | 意味が通じない。重大なエラーがある |

確認項目:
- 一文60文字以内か
- ですます調が統一されているか
- 驚き屋ワード（「やばい」「神」「革命」）が入っていないか
- 「正直に言うと」等のメタ前置きを入れていないか
- 機能一覧の羅列ではなく、読者の体験に沿って書いているか
- キャラの口調が concept.md に準拠しているか

### 軸5: 信頼性（Credibility）

このアカウントへの信頼を積み上げているか。

| Level | 基準 |
|-------|------|
| 5 | 独自の検証・実画面で他にない信頼を構築 |
| 4 | 一次ソースに基づき、制限事項も正直に記載 |
| 3 | 情報は正確だが、出典が不明瞭な箇所がある |
| 2 | 未検証の情報を断定している |
| 1 | 誤情報・誇張を含む |

確認項目:
- 一次ソース（公式ドキュメント等）を確認したか
- メリットだけでなく制限・デメリットも書いたか
- 「使ってみた」と書いているのに実際に使っていないことはないか
- 価格・スペック情報に「執筆時点（YYYY年M月）」を付けたか

## 共通レギュレーション

note・X 両方に適用するルール。

### テーマ選定の原則

**「誰でも知っていること」を声高に投稿しない。** 既知の常識をポストしても痛いだけ。

| NG | OK |
|----|-----|
| 有名すぎる話をそのまま紹介 | 難しい話を初学者にもわかるように噛み砕く |
| 「ChatGPTが便利！」（全員知ってる） | 「ChatGPTのこの機能、意外と知られてない」（知識差がある） |
| 周知の事実を大発見のように発信 | 専門的・ニッチな情報をミサキ層の言葉に翻訳 |

確認項目:
- このテーマは既にターゲット層の常識になっていないか
- 「知っている人が少ない」or「知っていても正しく使えていない」情報か
- 難しい話をわかりやすくする方向か、簡単な話を大げさにする方向か（後者はNG）
- **簡易判定**: AIの知識内だけで完結する話は価値が低い。Web検索が必要になるテーマほど価値が高い

### 一次解釈の原則

一次情報（公式発表）をそのまま並べるのではなく、**一次解釈**（実際に使っている人の目線で解釈した情報）を載せる。

| NG（一次情報の転記） | OK（一次解釈） |
|---------------------|---------------|
| 「自然言語でExcel操作が可能」 | 「VLOOKUPもSUMIFSも、日本語で伝えるだけ」 |
| 「スプレッドシートモデリング精度87%」 | 「精度87%。前世代から大幅に向上しています」+「執筆時点（2026年3月）」 |
| 「MCP経由でデータ連携」 | 「金融データ大手のムーディーズやダウ・ジョーンズのデータに直接アクセスできます」 |

- 公式の機能名をそのまま見出しに使わない。読者の体験に変換する
- 「できること」だけでなく「代わりに今日できること」（代替案）を入れる。現場で使っている人の手触り感が出る
- スペック → 体験の翻訳を常に行う

### 共通トーン

| ルール | 内容 |
|--------|------|
| 問題提起 | 読者が既に知っている悩みの描写は不要。ワクワク感で引く |
| 一般化した悩み | 「毎回ググってませんか？」等の共感取りに文字数を使わない |
| 驚き屋ワード | 禁止（やばい、神、革命、衝撃） |
| メタ前置き | 禁止（正直に言うと、ぶっちゃけ） |

## note記事レギュレーション

### トーンとスタイル

| ルール | 内容 |
|--------|------|
| 文体 | ですます調で統一 |
| リード | 全結論を入れる。新しい情報とワクワクで始める |
| 見出し | 【】フックは1箇所のみ。ダッシュ不使用。制限→欲求に変換 |
| 本文 | 情報パートは落ち着いたトーン。説明ベースで画像で崩す |
| 掛け合い | P8以外では味付け。導入 or 末尾で3往復以内 |

### 品質基準

| 項目 | 基準 |
|------|------|
| タイトル | 28〜32文字。ワクワク感 or 好奇心を起点にする |
| リード | 3行で全結論。「何がわかるか」を先出し |
| 見出し | 情報が一目でわかる素直な一文 |
| 図解 | 情報の可視化 + キャラで崩す。文字だけで伝わらない箇所に配置 |
| 参考リンク | 公式ソースのみ。記事末尾にまとめる |
| 文字数 | 2,000〜3,000文字（図解・掛け合い除く） |
| ハッシュタグ | 3〜5個。ミサキ層が検索しそうなワード |
| X誘導 | 記事末尾でXアカウントへの導線を入れる（「Xでも更新中」等。押し売りにならない自然な一文） |

## X投稿レギュレーション

### トーンとスタイル

| ルール | 内容 |
|--------|------|
| 文体 | 体言止め・短文。ですます不要 |
| 1行目 | 1秒で止まる。共感 or 意外性 |
| 情報密度 | 140文字で価値が完結。足りなければリプライスレッド |
| CTA | いいね・RT・リプライのどれを狙うか明確にする |

### 品質基準

| 項目 | 基準 |
|------|------|
| 本文 | 140文字以内で価値完結 |
| フック | プレスリリース調にしない。読者の体験に変換 |
| 画像 | 図解 or キャラ入り画像。テキストだけの投稿は避ける |
| ソース | note記事への導線（「詳細はプロフのnoteへ」）。URLは本文に入れない |
| スレッド | 2ツイート目以降で補足。1ツイート目だけで成立させる |
| 驚き屋NG | 「衝撃」「ヤバい」「神ツール」「革命」禁止 |
| メタ前置きNG | 「正直に言うと」「ぶっちゃけ」「本音」禁止 |
| フォロー誘導 | 週1回まで。押し売りにならない自然な文脈で |

## 驚き屋・AI臭プリフライトチェック

**Phase 2（執筆後）・Phase 3（校正前）で必ず実行する。**
生成テキスト全文に対して、以下のNGワードリストを機械的に照合する。1つでも該当したら修正必須。

### NGワード一覧

| カテゴリ | NGワード |
|---------|---------|
| 驚き屋 | やばい、ヤバい、ヤバすぎ、神、革命、衝撃、最強、究極、激変、爆速 |
| メタ前置き | 正直に言うと、ぶっちゃけ、本音、ガチ、本当のところ、正直、ここだけの話 |
| 大げさ修飾 | 圧倒的、劇的、驚異的、破壊的、常識を覆す、世界が変わる、もう戻れない |
| Before/After誇張 | 人生が変わった、完全に不要になった、もう手放せない、なしでは生きていけない |

### チェック手順

1. 下書きテキスト全文をスキャンし、上記NGワードに一致する箇所を列挙
2. 該当箇所ごとに「事実ベースの代替表現」に置換
3. 結果を校正Agentに引き渡す

### 置換パターン例

| NG表現 | 代替（事実ベース） |
|--------|-----------------|
| 「劇的に速くなった」 | 「体感で3倍くらい速い」 |
| 「もう手放せない」 | 「週3回は使っている」 |
| 「革命的な機能」 | 「これまでできなかったことができる」 |
| 「正直に言うと、すごい」 | 「使ってみたら、○○だった」 |

## 品質保証プロセス

```
0. 驚き屋・AI臭プリフライトチェック（上記NGワード照合）
1. god-writing チェックリスト（原則確認）
2. 3NOT対策チェック（読まない・信じない・行動しない）
3. 媒体別基準チェック（note or X）
4. committee_review（Upstream → Downstream → Executor）
5. ユーザーレビュー（Step H）
6. リリース
```
