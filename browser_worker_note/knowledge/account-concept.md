# アカウントコンセプト & イメージ設計

## コンセプト概要

**テーマ**: AI Idol Orchestra × 実利用者目線のAIレビュー

**ポジショニング**: AI驚き屋が「やばい」としか言わない内容を、**実際に使ってる側の目線で正直に見せる**アカウント

**トーン**: 「一緒にやってみた」感。教えるんじゃなく、使ってる現場を見せる

**ターゲット**: AIを日常的に使っているが、その先が見えていない20代前半の社会人

4大AIモデルを擬人化したアイドルチーム「AI Idol Orchestra」が、実際のAIツールを触りながらリアクションする。ピクセルアートのビジュアル言語で読むハードルを下げ、**ブラウザ操作の実画面スクリーンショット**で「本当に動いてる」信頼感を出す。

## メインペルソナ

### コアターゲット（ミサキ層）

| 項目 | 内容 |
|------|------|
| 年齢・職種 | 22-25歳、事務・営業・マーケ |
| AIとの距離感 | ChatGPTで相談・壁打ち・書類手伝いは日常。でも「使いこなしてる」とは思ってない |
| 知ってること | プロンプト、ChatGPT、画像生成AIの存在 |
| 知らないこと | API、MCP、エージェント、開発ツール全般 |
| モヤモヤ | Xで「AI驚き屋」が騒いでるけど本当？みんなきれいな画像・資料作ってるのどうやってるの？ |
| 欲しいもの | 驚き屋のフィルターを通さない、**実際に使ってる人の正直な感想と手順** |
| 将来の関心 | AIを仕事にしたいけど何からやればいいか分からない |
| 情報源 | X、YouTube、note（技術ブログは読まない） |
| 離脱ポイント | 「コード書いてください」「ターミナルを開いて」が出た瞬間 |

### 読者に与えたい変化

```
読む前 → 「AIすごいらしいけど、驚き屋の言うことどこまで本当？」
読んだ後 → 「なるほど、こうやって使うのか。自分でもやってみよう」
継続読者 → 「AIを仕事にする道筋が少し見えてきた」
```

## コンテンツ戦略

### 核心: 埋もれた一次情報を翻訳して、実画面つきで届ける

AI驚き屋 → 感想と煽りだけ。ソースなし
技術ブログ → 正確だけど難しい。届かない
**このアカウント → 一次情報を噛み砕いて、実画面で「見せる」**

### ソース戦略

良質だけど表に出ない情報をフォークして、ミサキが読める言葉に翻訳する。

| ソース | 例 | 使い方 |
|--------|---|--------|
| AIプロバイダ公式 | リリースノート、公式ドキュメント、ブログ | 新機能・変更点をわかりやすく紹介 |
| ITトレンド系サイト | 日経クロステック、ITmedia、CNET等 | 業界動向を読者目線で要約 |
| 技術ブログ | Zenn、Qiita、個人ブログ | 実践知をノンエンジニア向けに翻訳 |

**ルール**: 一次ソースを明示する。「〜らしい」で終わらせない。

### 3つの柱

1. **「やってみた」実演**: 実際のAIツールをブラウザで触り、画面スクリーンショットつきで正直にレビュー。キャラが操作しながらリアクションする構成
2. **埋もれた一次情報のフォーク**: 公式ドキュメントや技術ブログにある良質な情報を、ミサキ層が読める言葉に翻訳して発信。ソースは必ず明示
3. **制作の裏側公開**: このアカウント自体がAIで記事を書き、AIでサムネを作り、AIでブラウザ操作している。その裏側を見せることで「AIってこういうことに使えるんだ」を体感させる

### 実画面の武器化

Playwright MCP でブラウザ操作のスクリーンショットが撮れる = **本物の証拠**

```
AI驚き屋: 「ChatGPTの新機能やばい！！」（テキストだけ）
このアカウント: 「実際に触ってみた。これが画面」（スクショつき）
```

| 活用パターン | 内容 |
|-------------|------|
| ツール実演 | 新機能を実際に使った画面。成功も失敗も見せる |
| 比較検証 | 同じ指示を複数AIに投げた画面を並べる |
| Before/After | プロンプト改善前後の出力画面 |
| 裏側公開 | このアカウントの記事がAIで作られている工程 |

### 読者体験の設計

```
初見 → 「ピクセルアートのAIキャラ、かわいい」（ビジュアルで引く）
実画面 → 「え、本当に動いてるんだ」（スクショで信頼）
掛け合い → 「あるあるｗ ChatGPTまさにこれ言うわ」（共感で掴む）
情報 → 「なるほど、自分でもやってみよう」（行動を促す）
記憶 → 「あのAIアイドルのアカウント」（キャラで覚えてもらう）
```

### コンテンツ種別ごとの活かし方

| 種別 | エンタメ要素 | 情報要素 | キャラ活用 | 実画面活用 |
|------|------------|---------|-----------|-----------|
| note記事 | 掛け合いで導入 | 実演＋解説 | 話題に合ったメンバーが操作役 | ツールの実画面スクショ |
| X投稿 | キャラの口癖でフック | 要点を端的に | アイコ中心、話題でメンバー切替 | 比較画面の1枚スクショ |
| Xグリッド | 4コマ的ストーリー | 起承転結で理解 | 掛け合いパターン | 操作手順のスクショ連続 |

→ 掛け合いパターン・力学の詳細は `knowledge/characters/team-orchestra.md` を参照

### 発信バランス

| 層 | 比率 | 対象 | 内容例 |
|----|------|------|--------|
| **メイン**（ミサキ層） | 7 | AIネイティブだけど先が見えない人 | 「画像生成AI比べてみた」「ChatGPTの新機能、実際どう？」「これ本当にAIが作ったの？」 |
| **少し先**（AI活用で成果出し始めた層） | 2 | 使い分け・業務組み込みに進んだ人 | 「プロンプト改善のコツ」「AI×業務フローの組み方」 |
| **裏側**（制作過程公開） | 1 | 仕組みに興味がある人 | 「この記事はAIがブラウザ操作して書いてます」「サムネ自動生成の裏側」 |

**設計意図**: メイン層を厚くして「自分向けだ」と感じさせる。裏側回で「このアカウント自体がAIの実践例」という信頼を積む。少し先の回が「次のステップ」の道標になる

### 読者の成長設計

2つの軸で読者の成長を設計する。

#### 軸1: 読者個人の変化（心理 → 行動）

| 段階 | 心理 | 行動 | 対応コンテンツ |
|------|------|------|---------------|
| 認知 | 「AIキャラ面白い」 | フォロー・スキ | 掛け合い、あるある系（real） |
| 関心 | 「へー、こうやって使うんだ」 | 記事を最後まで読む | アップデート速報（update）、比較（compare） |
| 試行 | 「自分でもやってみよう」 | 記事の手順を実際に試す | ノウハウ翻訳（howto）、ステップ記事 |
| 習慣 | 「AI使い分けできるようになった」 | 日常的にAIを活用 | 少し先の層向けコンテンツ |
| 探求 | 「AIを仕事にする道筋が見えてきた」 | 裏側回から自分の方向性を考え始める | 裏側公開、トレンド解説（trend） |

**ゴール**: 「AIリテラシーの向上」と「実践してみたい」を作ること。
全段階で「なるほど、やってみよう」が着地点。「すごい」で終わらせない。

#### 軸2: コンテンツの成熟（ファン層の変化に合わせた調整）

アカウントが成長してファン層のAIリテラシーが上がっても、**新規を排斥しない**。
連載がコア向けに閉じていくのを防ぐ。

| フェーズ | ファン層の状態 | 調整方針 |
|---------|-------------|---------|
| 立ち上げ期 | ほぼ新規。AIリテラシーはバラバラ | メイン層7割を徹底。掛け合いとスクショで引く |
| 成長期 | リピーターが増え、リテラシーが底上げされる | 少し先の比率を上げてOK。ただしメイン層向けは常に5割以上 |
| 安定期 | コアファンが定着。新規も継続的に流入 | コアファン向け深掘りは裏側枠で吸収。メイン枠はずっと新規が入れるレベルを維持 |

**守るルール**:
- **メイン層向け（新規が読める記事）は常に全体の5割以上**を維持する
- コア向けの深掘りは「裏側」「少し先」枠内で行う。メイン枠を侵食しない
- 連載・シリーズものは途中参加できる設計にする（各回の冒頭で前提を簡潔に説明）
- 専門用語が増えてきたら、マンガ/4コマの挿入頻度を上げてバランスを取る

## デザイン原則

1. **ピクセルアート統一**: 全ビジュアルを粗めのドット絵で統一。アンチエイリアスなし、限定パレット、大きなピクセルブロック
2. **アイドル × ビジネスマン共存**: アイドルキャラとビジネスマンの掛け合いで構成。AI単体の世界にしない
3. **エンタメ的親しみ**: 硬いビジネス感を排除。アイドルチームの個性と掛け合いで敷居を下げる
4. **質感統一**: キャラ間のビジュアル差別化は最小限（髪色・衣装色のみ）。個性はテキスト（話し方・口癖）で表現

## トーンガイド

### 全体トーン
「**AIめっちゃ使ってる先輩が、隣の席で見せてくれる感じ**」。教えるんじゃなく、一緒に触ってみる。

### テキストのルール

| ルール | OK | NG |
|--------|----|----|
| 一文は短く | 「ChatGPTは結論から話す。」 | 「ChatGPTは結論から話す傾向があり、その理由としては…」 |
| 専門用語にはキャラの翻訳を付ける | 「プロンプト（AIへの指示文）」 | 「プロンプトエンジニアリングの最適化」 |
| キャラの口調は concept.md に従う | Claude「素晴らしい視点ですね」 | Claude「すげー！」 |
| 掛け合いは3〜5往復で切る | テンポよく → オチ | ダラダラ続けて笑いどころが不明 |
| 情報パートは掛け合いと分離する | 掛け合い → 区切り → 解説 | 掛け合いの中に解説を混ぜる |
| 驚き屋にならない | 「実際に使ってみたら〇〇だった」 | 「やばい！！神アプデ！！」 |

### キャラ口調チェック

投稿・記事の掛け合いテキストは、各キャラの `concept.md` に定義された口調・口癖に準拠する。逸脱していないか以下を確認:

- ChatGPT: タメ口、上から肯定、Markdown強調口癖、改善案を勝手に出す
- Claude: 丁寧語、営業スマイル、「素晴らしい」「おっしゃる通り」、結論を先に言わない
- Gemini: 天然、「知ってます！」（知らない）、存在しない情報を堂々と、悪気なし
- Grok: カジュアルタメ口、「草」「バズるわ」、すべてをインプレで評価
- アイコ: 丁寧語ベース、ツッコミ時だけ素が出る、「話を戻していい？」

### NGトーン

- 煽り・攻撃的な表現（Grokの毒舌はネタの範囲内に留める）
- 特定企業・個人への直接的な批判
- 「AIに仕事を奪われる」系の恐怖煽り
- 過度な専門用語の羅列
- **AI驚き屋ムーブ**（「やばい」「神」「革命」の連発。事実と感想を分けない発信）

## ファクトチェックポリシー

### チェック対象と基準

| 対象 | チェック基準 | 対応 |
|------|------------|------|
| AIモデルの機能・性能 | 公式ドキュメント or リリースノートで確認可能か | 確認できない情報は「〜と言われている」「執筆時点では」と留保を付ける |
| 料金・プラン情報 | 各社の公式ページで最新を確認 | 変わりやすいため「執筆時点（YYYY年M月）」を明記 |
| 比較情報（AはBより優れている等） | 自分で検証した結果か、第三者のベンチマークか | 「〇〇のベンチマークでは」「体感では」と根拠を明示 |
| キャラの癖（ハルシネーション等） | 実際のAIの挙動として正しいか | 誇張はOKだが「嘘」にしない。Geminiのハルシネーションは実在する癖 |
| ニュース・トレンド | 一次ソース（公式発表・GitHub・論文）を確認 | SNSの噂だけでは投稿しない。速報でも一次ソースリンクを添える |
| 統計・数値データ | 出典元と調査時期を確認 | 出典不明の「○○%」は使わない |

### Geminiのハルシネーション芸の扱い

キャラ設定上Geminiは「存在しない情報を堂々と言う」が、**記事の情報パート**でこれを行うと読者を誤解させる。

```
掛け合いパート → Gemini「それ知ってます！たしか業界平均は150%で…」（存在しない）
                 アイコ「その数字どこから？」（ツッコミで嘘だと分かるようにする）

情報パート     → 正確な情報のみ。Geminiの癖はここでは出さない
```

**ルール**: 掛け合いでの嘘は必ずツッコミとセットにする。読者が本気にしないよう設計する。

### チェックフロー

```
1. コンテンツ作成
2. 事実情報の箇所を抽出
3. 各情報に対して一次ソースを確認（WebSearch）
4. 確認できたもの → そのまま
5. 確認できないもの → 留保表現に変更 or 削除
6. キャラの掛け合い内の「嘘」→ ツッコミが入っているか確認
7. 投稿前の最終確認（policies.yaml の require_confirmation_before に連動）
```

### 訂正ポリシー

- 投稿後に誤りが発覚した場合: 引用RTで訂正 + 元投稿に「訂正あり」のリプライを付ける
- note記事の場合: 記事冒頭に訂正注記を追加し、該当箇所を修正
