# アカウントコンセプト & イメージ設計

## コンセプト概要

**テーマ**: AI Idol Orchestra × ビジネス活用

**トーン**: 親しみやすさ・エンタメ感・テック感の融合

**ターゲット**: 業務でAIを使い始めた／使いたい現場の人。採用広報・社内教育の素材としても使える層

4大AIモデルを擬人化したアイドルチーム「AI Idol Orchestra」が、ビジネスマンと掛け合いながらAIの扱い方を解説する。ピクセルアートのビジュアル言語で読むハードルを下げ、「AIを使ってみたいけど何から始めれば？」という人でも気軽に読める導線を作る。

### 想定読者

| 読者タイプ | ニーズ | コンテンツの刺さり方 |
|-----------|--------|-------------------|
| AI初心者（使い始めた人） | どう指示すればいいか分からない | キャラの癖＝AIの癖を知って「こう頼めばいいのか」が分かる |
| 活用中の現場担当 | 使い分け・プロンプトの精度を上げたい | 掛け合いで各AIの得意不得意が自然に入る |
| 採用広報・教育担当 | 社内にAIリテラシーを広めたい | エンタメ性があるので研修素材・社内共有しやすい |
| AI開発・エージェント関心層 | 最新ツールの実践知やトレンドを追いたい | 「ガチ回」の深掘り記事で刺さる。普段回のエンタメで定着 |

---

## コンテンツ戦略

### 核心: 「AIの癖」をキャラの個性にして、有益な情報をエンタメで届ける

読者に「**この人、各AIの特徴をちゃんと押さえてるな**」と思わせつつ、掛け合いのエンタメ性で読みやすくする。AI生成コンテンツであることを隠さず、むしろ**AIの癖を大げさにキャラ化**して価値に変える逆張り。

### 4つの柱

1. **AI同士の掛け合い**: 各AIの実際の癖（ChatGPTの上から肯定、Claudeの営業スマイル、Geminiのハルシネーション、Grokのバズ脳）を誇張してぶつける。会話がかみ合わないのが笑いの核
2. **現場で使えるAIの扱い方**: 「このAIにはこう指示すると上手くいく」という実践ノウハウを、キャラの個性を通して自然に伝える。比較記事ではなく掛け合いで差が分かる
3. **AI事業の報告・ニュース・トレンド**: AIエージェント（Claude Code、Codex、Cursor等）の実践、MCPサーバーの活用法、何が手動で何が自動化できるのか——業界の動きや開発現場のリアルをエンタメで発信。たまに出る「ガチ回」として深掘り記事も
4. **AI生成コンテンツの逆張り**: 「AIっぽい文章」を欠点ではなくキャラの個性として活用。AI生成だからこそ面白いコンテンツを作る

### 読者体験の設計

```
初見 → 「ピクセルアートのAIキャラ、かわいい」（ビジュアルで引く）
掛け合い → 「あるあるｗ ChatGPTまさにこれ言うわ」（共感で掴む）
情報 → 「なるほど、こう指示すればいいのか」（実用価値を届ける）
記憶 → 「あのAIアイドルのアカウント」（キャラで覚えてもらう）
```

### コンテンツ種別ごとの活かし方

| 種別 | エンタメ要素 | 情報要素 | キャラ活用 |
|------|------------|---------|-----------|
| note記事 | 掛け合い・4コマで導入 | 本文で解説 | 話題に合ったメンバーが解説役 |
| X投稿 | キャラの口癖でフック | 要点を端的に | アイコ中心、話題でメンバー切替 |
| Xグリッド | 4コマ的ストーリー | 起承転結で理解 | 掛け合いパターンをそのまま使う |
| クイズ | キャラがプレゼンター | 知識確認・議論誘導 | アイコ or 話題に合ったメンバー |

→ 掛け合いパターン・力学の詳細は `knowledge/characters/team-orchestra.md` を参照

### 発信バランス（難易度別の頻度配分）

| レベル | 比率 | 対象読者 | 内容例 |
|--------|------|---------|--------|
| 初心者向け | 3 | AI使い始めた人 | 「ChatGPTへの頼み方」「AIに丸投げしていいこと・ダメなこと」「プロンプトの基本」 |
| 中級向け | 2 | 活用中の現場担当 | 「AI使い分けガイド」「プロンプト改善テクニック」「業務フロー×AI組み込み」 |
| 上級向け（ガチ回） | 1 | 開発者・エージェント関心層 | 「Claude Codeで何が自動化できるか」「MCPサーバー活用」「AIエージェント開発のリアル」 |

**設計意図**: 初心者向けを厚くして間口を広げつつ、たまに出るガチ回で「この人ちゃんと開発もやってるんだ」という信頼感を積む。上級回が刺さった読者は過去の初心者回も遡って読む導線になる

## デザイン原則

1. **ピクセルアート統一**: 全ビジュアルを粗めのドット絵で統一。アンチエイリアスなし、限定パレット、大きなピクセルブロック
2. **アイドル × ビジネスマン共存**: アイドルキャラとビジネスマンの掛け合いで構成。AI単体の世界にしない
3. **エンタメ的親しみ**: 硬いビジネス感を排除。アイドルチームの個性と掛け合いで敷居を下げる
4. **質感統一**: キャラ間のビジュアル差別化は最小限（髪色・衣装色のみ）。個性はテキスト（話し方・口癖）で表現

## トーンガイド

### 全体トーン
「**友達のIT担当が教えてくれる感じ**」。専門用語を避けるのではなく、キャラの口癖で噛み砕く。

### テキストのルール

| ルール | OK | NG |
|--------|----|----|
| 一文は短く | 「ChatGPTは結論から話す。」 | 「ChatGPTは結論から話す傾向があり、その理由としては…」 |
| 専門用語にはキャラの翻訳を付ける | 「プロンプト（AIへの指示文）」 | 「プロンプトエンジニアリングの最適化」 |
| キャラの口調は concept.md に従う | Claude「素晴らしい視点ですね」 | Claude「すげー！」 |
| 掛け合いは3〜5往復で切る | テンポよく → オチ | ダラダラ続けて笑いどころが不明 |
| 情報パートは掛け合いと分離する | 掛け合い → 区切り → 解説 | 掛け合いの中に解説を混ぜる |

### キャラ口調チェック

投稿・記事の掛け合いテキストは、各キャラの `concept.md` に定義された口調・口癖に準拠する。逸脱していないか以下を確認:

- ChatGPT: タメ口、上から肯定、Markdown強調口癖、改善案を勝手に出す
- Claude: 丁寧語、営業スマイル、「素晴らしい」「おっしゃる通り」、結論を先に言わない
- Gemini: 天然、「知ってます！」（知らない）、存在しない情報を堂々と、悪気なし
- Grok: カジュアルタメ口、「草」「バズるわ」、すべてをインプレで評価
- アイコ: 丁寧語ベース、ツッコミ時だけ素が出る、「話を戻していい？」

### NGトーン

- 煽り・攻撃的な表現（Grokの毒舌はネタの範囲内に留める）
- 特定企業・個人への直接的な批判
- 「AIに仕事を奪われる」系の恐怖煽り
- 過度な専門用語の羅列（ガチ回でも平易な導入を付ける）

## ファクトチェックポリシー

### チェック対象と基準

| 対象 | チェック基準 | 対応 |
|------|------------|------|
| AIモデルの機能・性能 | 公式ドキュメント or リリースノートで確認可能か | 確認できない情報は「〜と言われている」「執筆時点では」と留保を付ける |
| 料金・プラン情報 | 各社の公式ページで最新を確認 | 変わりやすいため「執筆時点（YYYY年M月）」を明記 |
| 比較情報（AはBより優れている等） | 自分で検証した結果か、第三者のベンチマークか | 「〇〇のベンチマークでは」「体感では」と根拠を明示 |
| キャラの癖（ハルシネーション等） | 実際のAIの挙動として正しいか | 誇張はOKだが「嘘」にしない。Geminiのハルシネーションは実在する癖 |
| ニュース・トレンド | 一次ソース（公式発表・GitHub・論文）を確認 | SNSの噂だけでは投稿しない。速報でも一次ソースリンクを添える |
| 統計・数値データ | 出典元と調査時期を確認 | 出典不明の「○○%」は使わない |

### Geminiのハルシネーション芸の扱い

キャラ設定上Geminiは「存在しない情報を堂々と言う」が、**記事の情報パート**でこれを行うと読者を誤解させる。

```
掛け合いパート → Gemini「それ知ってます！たしか業界平均は150%で…」（存在しない）
                 アイコ「その数字どこから？」（ツッコミで嘘だと分かるようにする）

情報パート     → 正確な情報のみ。Geminiの癖はここでは出さない
```

**ルール**: 掛け合いでの嘘は必ずツッコミとセットにする。読者が本気にしないよう設計する。

### チェックフロー

```
1. コンテンツ作成
2. 事実情報の箇所を抽出
3. 各情報に対して一次ソースを確認（WebSearch）
4. 確認できたもの → そのまま
5. 確認できないもの → 留保表現に変更 or 削除
6. キャラの掛け合い内の「嘘」→ ツッコミが入っているか確認
7. 投稿前の最終確認（policies.yaml の require_confirmation_before に連動）
```

### 訂正ポリシー

- 投稿後に誤りが発覚した場合: 引用RTで訂正 + 元投稿に「訂正あり」のリプライを付ける
- note記事の場合: 記事冒頭に訂正注記を追加し、該当箇所を修正
