# コンテンツタイプ定義

記事は **内容（C）× 構成パターン（P）× 加工方法（T）** の3軸で設計する。
演出（サムネスタイル・キャラ構成・掛け合いの有無）はさらに独立したレイヤー。

## 共通ルール

- **リードに全結論を入れる**。読者は冒頭で読む価値を判断する
- 結論を隠して最後まで引っ張る構成は使わない

---

## 構成パターン（P）— 記事の骨格

| ID | パターン | 骨格 | 特徴 |
|----|---------|------|------|
| P1 | PREP | 結論→理由→具体例→結論 | 最も汎用。主張を伝えたい記事に |
| P2 | ステップ | 結論→手順1→手順2→…→まとめ | 再現性が命。1ステップ=1アクション |
| P3 | 並列比較 | 結論→比較表→軸ごとの詳細→向き不向き | 選択を助ける。同一基準で統一 |
| P4 | 逆三角形 | 最重要ファクト→背景→詳細→補足 | 速報・ニュース向き。途中離脱されても成立 |
| P5 | リスト | 結論→項目1→項目2→…→総括 | ○選・ランキング。数字タイトルでCTR↑ |
| P6 | Before/After | 結論→問題の状態→やったこと→変化後→学び | 変化の落差が価値。体験記と相性◎ |
| P7 | Q&A | 結論→Q1→A1→Q2→A2→… | 読者の疑問を先回り。検索意図に直結 |
| P8 | 対話劇 | 結論→キャラA発言→B反応→…→まとめ | 掛け合いで自然に情報伝達。うちの主力形式 |
| P9 | ケーススタディ | 結論→状況→施策→結果→分析→教訓 | 事例ベース。具体性と再現性の両立 |
| P10 | Problem/Solution | 結論→問題定義→原因分析→解決策→実行例 | 課題駆動。読者の痛みから入る |

## 内容（C）— 何について書くか

| ID | 内容 | 例 |
|----|------|-----|
| C1 | ツール紹介 | Claude Code、ChatGPT、Gemini等の機能紹介 |
| C2 | ノウハウ | プロンプトの書き方、AI活用術 |
| C3 | ニュース | 新モデルリリース、API変更、業界動向 |
| C4 | 体験 | 実際にやってみた結果・学び |
| C5 | 用語 | AI用語、技術概念の解説 |
| C6 | 比較 | AI同士、ツール同士の比較検証 |
| C7 | オピニオン | AI活用への見解、業界への提言 |
| C8 | トレンド分析 | データや動向の読み解き |
| C9 | AI歴史 | AIの転換点・裏話・制度変遷の振り返り |

## 加工方法（T）— 情報をどう処理するか

C（何を書くか）と P（どう構成するか）が決まっても、**情報の加工方法**が違えば記事の印象は大きく変わる。T軸は「集めた情報をどう料理するか」を定義する。

| ID | 加工方法 | 一言定義 | 向いているC | OODAとの対応 |
|----|---------|---------|-----------|-------------|
| T1 | 情報整理 | 散在する事実をまとめて構造化する | C3, C5 | Observe重視 |
| T2 | 情報翻訳 | 専門情報をミサキ層の言葉に変換する | C5, C8 | Orient重視 |
| T3 | 情報解釈 | 事実に「だからこう読める」を加える | C7, C8 | Detect重視 |
| T4 | 検証 | 「本当？」を実際に確かめる | C1, C4 | Detect+Interpret |
| T5 | 実践 | やってみた結果を見せる | C1, C2 | Interpret重視 |
| T6 | 比較 | 同じ基準で並べて違いを出す | C6 | Orient+Detect |
| T7 | 対比 | 常識と事実のギャップを際立たせる | C8, C9 | Orient→Detect |

### T の選び方（5軸ベクトルから）

リサーチ情報の5軸分類（`generate-content.md` Phase 1-2.5 参照）に基づいて選ぶ。

| 5軸の傾向 | 推奨T | 理由 |
|----------|-------|------|
| awareness 低 + discovery 鋭い | **T7 対比** | 常識を覆す |
| awareness 中 + forgotten 強い | **T3 解釈** | 知ってるつもりを更新 |
| psychological 強い | **T2 翻訳** | 難しい話を噛み砕く |
| discovery が複数 | **T1 整理** or **T6 比較** | 情報量を活かす |
| semantic が実行可能 | **T5 実践** | 手を動かして見せる |

### チャレンジ回ローテーション

直近3記事の T を `knowledge/logs/` で確認する。同じ T が2回続いた場合、3回目はその T を選ばず、使用頻度の低い T を優先選択する。

### C×P×T の掛け合わせ例

| 内容 × 構成 × 加工 | 記事イメージ |
|-------------------|-------------|
| C3 × P4 × T7 | 「Notion AIの中身がChatGPTじゃなかった件」（常識と事実の対比） |
| C2 × P2 × T5 | 「AIに議事録を任せる方法【実際にやった3ステップ】」（実践） |
| C8 × P1 × T3 | 「AIツールの裏でモデルが入れ替わっている、という読み解き」（解釈） |
| C6 × P3 × T6 | 「ChatGPT vs Claude、同じ指示を出して比べた」（比較） |
| C5 × P7 × T2 | 「MCPって何？難しそうだけど要はこういうこと」（翻訳） |
| C1 × P6 × T4 | 「精度87%って本当？実際に試してみた」（検証） |
| C9 × P4 × T1 | 「GPT-3からGPT-4oまで。何が変わったか整理」（情報整理） |

## 構成パターン詳細

### P1: PREP

```
リード: 結論（主張）を明示
H2: 理由 — なぜそう言えるのか
H2: 具体例 — データ・事例・体験で裏付け
  （想定反論 → 再反論を挟むと厚みが出る）
H2: まとめ — 結論の再強調 + 展望
```

### P2: ステップ

```
リード: 結論（何ができるようになるか）+ ステップ数を明示
H2: ステップ1 — ○○する
  H3: やり方 + 注意点（図解1枚以上）
H2: ステップ2 — ○○する
  ...
H2: FAQ（必要なら）
H2: まとめ — 手順の箇条書き要約
```
- ステップ数は 3〜7（5前後が最適）
- 各ステップに画像/図解を1枚以上

### P3: 並列比較

```
リード: 結論（どちらが何に向いているか）
H2: 比較表（サマリー）← 先出し
H2: 比較軸1の詳細
H2: 比較軸2の詳細
  ...
H2: こんな人にはA / こんな人にはB
H2: まとめ
```
- 比較軸を事前に確定してからデータ収集
- 全対象に同一構成・同一基準を適用

### P4: 逆三角形

```
リード: 最重要ファクト（5W2H）
H2: 背景・経緯
H2: 詳細データ・関係者の声
H2: 影響・今後の展望
H2: 自分たちの見解
```
- 重要度の高い順に配置。途中で読むのをやめても情報が成立する

### P5: リスト

```
リード: 結論 + 総数を明示
H2: 項目1 — 名前
  H3: 特徴 / メリット / 注意点
H2: 項目2 — ...
  ...
H2: 比較表（サマリー）
H2: まとめ + CTA
```
- 全項目のフォーマットを統一
- 1項目 100〜200語
- 数字入りタイトルでCTR +36%

### P6: Before/After

```
リード: 結論（変化の要約）
H2: Before — 問題の状態（痛みを具体的に）
H2: やったこと — 施策・行動
H2: After — 変化後の状態（数値で示す）
H2: 学び — 再現可能な教訓
```
- 変化の落差が大きいほど価値が高い
- 五感・定量表現で臨場感を出す

⚠️ **驚き屋ドリフト注意**: P6は変化を強調する構成のため、驚き屋論法にドリフトしやすい。以下に該当したらP4（逆三角形）への変更を検討する:
- Before/Afterの差を過剰に演出している（「量産」「別の生き物のように」）
- 変化の原因が曖昧なまま結果だけ強調している
- 読者に「自分もこうなれる」と過度な期待を持たせている
- このアカウントのトーン（淡々と事実を見せる）と合っていない

### P7: Q&A

```
リード: 結論（このテーマの核心を一言で）
H2: Q1. よくある質問
  A1. 回答
H2: Q2. ...
  A2. ...
  ...
H2: まとめ
```
- 読者の疑問を先回りして構成
- FAQ Schema 対応で検索結果にリッチスニペット表示

### P8: 対話劇

```
リード: 結論（この掛け合いで何がわかるか）
H2: 導入シーン — テーマ提示
  キャラA: 発言
  キャラB: 反応（ボケ or ツッコミ）
  ...
H2: 展開シーン — 情報の核心
  （キャラの個性フィルターを通して情報を伝える）
H2: オチ — まとめ or 意外な結論
H2: 今日のポイント — 箇条書きで情報を整理
```
- キャラのフィルター（`characters/*/concept.md`）を通して情報伝達
- 掛け合い部分と情報整理部分を分離する

### P9: ケーススタディ

```
リード: 結論（事例から得られた教訓）
H2: 状況 — 何がどうなっていたか
H2: 施策 — 何をしたか
H2: 結果 — どうなったか（数値）
H2: 分析 — なぜそうなったか
H2: 教訓 — 再現可能な学び
```
- 具体性と再現性の両立
- 数値データを必ず含める

### P10: Problem/Solution

```
リード: 結論（問題と解決策の要約）
H2: 問題定義 — 読者の痛みを言語化
H2: 原因分析 — なぜその問題が起きるのか
H2: 解決策 — 具体的な対処法
H2: 実行例 — 解決策を適用した結果
H2: まとめ
```
- 読者の課題から入る構成
- 原因→解決策の因果関係を明確に

## 演出ガイド — 何をいつ挟むか

演出は構成パターン（P）とは独立したレイヤー。以下のルールで挿入する。

**大原則: 本文（情報パート）は落ち着いたトーンで書く。**
掛け合い・マンガはあくまで導入やブレイクの役割。情報パートに入ったらエンタメ感を抑え、読者が内容に集中できるようにする。

### 演出要素一覧

| 演出 | 用途 | 挿入条件 |
|------|------|---------|
| 掛け合い | エンタメ・共感・導入 | 全記事で使える。ただし本体ではなく味付け（P8 除く） |
| 実画面スクショ | 信頼・証拠・手順図解 | 実演判定OKのネタのみ |
| 図解 | 比較・構造・フローの可視化 | 文章だけでは伝わりにくい時 |
| マンガ/4コマ | 難しい概念・用語の噛み砕き | 専門用語や抽象概念の解説時 |

### 構成パターン別の演出挿入マップ

| パターン | 掛け合い | 実画面スクショ | 図解 | マンガ |
|---------|---------|---------------|------|--------|
| P1 PREP | 導入で1回 | 具体例セクションに | 理由の構造化に | - |
| P2 ステップ | 導入で1回 | 各ステップに必須（実演時） | 手順フローに | 前提知識の解説に |
| P3 並列比較 | 導入 or まとめで1回 | 各AIの出力画面を並べる | 比較表として | - |
| P4 逆三角形 | 末尾のリアクションで1回 | 冒頭に新機能の画面 | 変更点の図解に | 新用語の解説に |
| P5 リスト | 導入で1回 | 各項目のUI画面 | 比較表に | - |
| P6 Before/After | 導入で1回 | Before画面 + After画面 | 変化の図解に | - |
| P7 Q&A | 各Q&Aをキャラに振る | 回答の裏付けに | 回答の図解に | 難しいQの噛み砕きに |
| P8 対話劇 | **本体**（3-5往復） | 掛け合い中の「見て」で挿入 | 今日のポイントに | 難しい概念の解説に |
| P9 ケーススタディ | 導入で1回 | 施策・結果の実画面 | 分析の図解に | - |
| P10 Problem/Solution | 導入で問題を体感させる | 解決策の実行画面 | 原因分析の図解に | 問題の概念説明に |

### 掛け合いの挿入ルール

- P8（対話劇）以外では**味付け**。本文を圧迫しない
- 導入の掛け合いは **3往復以内** で切る
- 掛け合いの後に必ず区切りを入れてから情報パートに入る
- 掛け合い内の情報は「きっかけ」程度。詳細は情報パートで

### 実画面スクショの挿入ルール

- 実演判定OK（`.claude/workflows/research.md` 参照）の場合のみ
- 1記事あたり **3〜8枚** が目安。多すぎると読みにくい
- キャプション必須。「何の画面で、何を見ればいいか」を1文で
- Before/After は必ずセットで

### マンガ/4コマの挿入ルール

- 以下のタイミングで使う。全記事に入れるわけではない
  - **難しい単語・概念の解説**: 「API」「トークン」等、ミサキ層が離脱しそうな用語が出た時
  - **長文が続いた後のブレイク**: 情報パートが長くなった時にリズムを変える
- キャラが「え、それ何？」→ 別のキャラが噛み砕いて説明 → 「そういうことか！」の3コマ構成
- 1記事に1〜2箇所が上限。入れすぎると情報密度が下がる

### 図解の挿入ルール

- 以下の場面で使う:
  - **抽象概念の可視化**: 文章だけでは掴みにくい概念を図で示す
  - **まとめ・要約**: セクション末や記事末で情報を整理する
  - **ランキング・比較表**: 順位や比較を視覚的に
  - **ソースデータのグラフ化**: 数値・統計を読みやすく変換
- キャラは図解内には入れない（情報の邪魔になる）
- シンプルに。色はカラーパレット（account-concept.md）に従う

## 参考情報の扱い

### 掲載ルール

- **記事にリンクを載せるのは公式ソースのみ**
  - AIプロバイダ公式サイト、公式ブログ、公式ドキュメント、公式GitHub
- 技術ブログ・個人ブログは翻訳・参考の元ネタとして使うが、記事にリンクは貼らない
- ITトレンド系メディアも記事にリンクは貼らない

### 掲載フォーマット

```
記事末尾に「参考」セクションを設ける:

---
## 参考
- [Anthropic — Claude 4.5 リリースノート](https://anthropic.com/...)
- [OpenAI — GPT-Image-1.5 ドキュメント](https://platform.openai.com/...)
```

### 本文中の言及

- 「公式ドキュメントによると〜」「リリースノートでは〜」と自然に言及する
- URLは本文中に直接書かない。末尾の参考セクションに集約
- 「執筆時点（YYYY年M月）」を価格・スペック情報に付ける
