# /generate-content ワークフロー

リサーチハンドオフ（または直接テーマ指示）を受け取り、**下書きファイルを中心に磨き上げてから**投稿する。

## 下書きファースト原則

記事は必ず `knowledge/drafts/` にMarkdownファイルとして作成し、磨き上げてからnote/Xに投稿する。
**下書きファイルが全工程の中心成果物。** ブラウザ投稿は最後の転記作業にすぎない。

```
knowledge/drafts/
├── note_{スラッグ}.md      ← note記事の下書き
├── x_{スラッグ}.md         ← X投稿の下書き
└── ...
```

### 下書きファイルの構造

```markdown
# 下書き: {タイトル}

- **C×P×T**: {C×P×T}
- **ステータス**: 下書き | 検証済み | 校正済み | 投稿済み
- **想定文字数**: {N}〜{N}文字
- **ハッシュタグ**: #{tag1} #{tag2}
- **タイトル文字数**: {N}文字

## 画像

| # | 種別 | ファイル | 挿入位置 | 検証 |
|---|------|---------|---------|------|
| 1 | サムネイル | `knowledge/data/images/note_thumbnail_{slug}_{YYYY-MM-DD}.jpg` | 記事トップ | [ ] |
| 2 | ソーススクショ | `knowledge/data/images/source_{slug}_01.png` | H2-xの下 | [ ] |
| 3 | 図解 | `knowledge/data/images/note_diagram_{slug}_{YYYY-MM-DD}.jpg` | H2-xの下 | [ ] |

---

{リード}

---

{本文}

---

## 参考

- [ソース名](URL)
```

### 画像の種別と役割

| 種別 | 用途 | 生成コマンド | 必須度 |
|------|------|------------|--------|
| サムネイル | 記事トップ画像（1280x670px） | `python scripts/generate_image.py thumbnail "{説明}" --topic {slug}` | note記事は必須 |
| ソーススクショ | 一次ソースの証拠画像 | Playwright `browser_take_screenshot` | C1必須、C3/C8/C9推奨 |
| 図解 | 概念の可視化・比較表 | `python scripts/generate_image.py diagram "{説明}" --topic {slug}` | 任意（長文のブレイクに有効） |
| 図解（ボード型） | フロー図・比較表・手順リスト | `python scripts/generate_image.py diagram "{説明}" --topic {slug} --layout board` | 任意 |
| 実演スクショ | やってみた系の実画面 | Playwright `browser_take_screenshot` | C1必須 |

### コマンドリファレンス

各フェーズで使うコマンドを統一する。コマンド経由で実行すればプロンプトログが自動記録される。

```bash
# --- サムネイル生成 ---
python scripts/generate_image.py thumbnail "AIアイドルがオフィスで働く風景" --topic chatgpt-history
python scripts/generate_image.py thumbnail "黒板でAI解説" --topic digital-ai-subsidy --style blackboard
python scripts/generate_image.py thumbnail "VSバトル構図" --topic gemini-vs-claude --style vs-battle --chars gemini,claude

# --- 図解生成 ---
python scripts/generate_image.py diagram "プロンプトの3つのコツ" --topic prompt-tips
python scripts/generate_image.py diagram "AIコスト比較表" --topic ai-cost --layout board

# --- 4コマ漫画 ---
python scripts/generate_image.py comic "AIに議事録を任せた結果" --topic meeting-ai --story "1:残業の山|疲れた表情 2:AI発見！|驚く 3:全自動！|お茶 4:...え？|文字化け"

# --- クロップ（既存画像のリサイズ） ---
python scripts/generate_image.py crop knowledge/data/images/source.jpg --preset note-thumbnail

# --- ソーススクショ（Playwright MCP） ---
# 1. browser_navigate → URL遷移
# 2. browser_snapshot → 構造確認
# 3. browser_take_screenshot → PNG保存
# 保存先: knowledge/data/images/source_{slug}_{連番}.png
```

**ログ**: 全生成コマンドのプロンプト・パラメータが `knowledge/logs/image-generation/prompt-log.jsonl` に自動記録される。キャラ揺れ等の原因追跡に使用。

### ステータス遷移

```
下書き → 検証済み → 校正済み → 投稿済み
  ↑         |          |
  └─ 差し戻し ←─────────┘
```

| ステータス | 意味 | 次のアクション |
|-----------|------|---------------|
| 下書き | Phase 2 完了。本文あり | Phase 2.5（検証）へ |
| 検証済み | ファクトチェック・スクショ・具体性OK | Phase 3（校正）へ |
| 校正済み | 品質 Level 4+ 確認済み | ユーザー承認 → Phase 4（投稿）へ |
| 投稿済み | noteまたはXに公開完了 | URL記録・ログ更新 |

### セッション分割の標準パターン

記事の作成と投稿は**別セッションでもよい**。下書きファイルが状態を保持するので、途中で中断しても次のセッションで再開できる。

```
セッション A: テーマ指示 → Phase 1〜2 → 下書き作成
セッション B: 下書き Read → Phase 2.5〜3 → 磨き上げ
セッション C: 校正済み下書き → Phase 4 → note投稿
```

**複数記事のバッチ作成**: 下書きを先に全部作り、まとめて検証・校正するのも有効。ただし執筆は1記事ずつ順番に行う（並列執筆は品質が崩れる）。

## 使い方

```
/generate-content <テーマ or ハンドオフ> [媒体: note|x] [C×P×T指定]
/generate-content --resume <draft_path>              ← 既存下書きから再開
/generate-content --resume <draft_path> --phase 2.5  ← 指定フェーズから再開
```

例:
- `/generate-content "ChatGPT for Excel" 媒体: note C3×P4`
- `/generate-content handoff: knowledge/logs/research/2026-03-12.md#candidate-1`
- `/generate-content --resume knowledge/drafts/note_chatgpt-history.md`
- `/generate-content --resume knowledge/drafts/note_chatgpt-history.md --phase 3`

### --resume モード

既存の下書きファイルから途中再開する。下書きのステータスから開始 Phase を自動判定する。

| 下書きステータス | 自動開始 Phase | やること |
|----------------|---------------|---------|
| 下書き | Phase 2.5 | 検証から |
| 検証済み | Phase 3 | 校正から |
| 校正済み | Phase 4 | 投稿へ |

`--phase` で明示指定すると自動判定を上書きする。Phase 1 を指定するとアウトラインから作り直す（下書きは上書きされる）。

## エージェント人格定義（Big Five × DISC）

サブエージェントには「役割」ではなく「人格」を与える。Big Five（人格特性）で思考・価値観を、DISC（タスク行動）で仕事の進め方を定義する。スコアは 1（低）〜 5（高）。

### 見出しAgent

| 軸 | スコア | 行動指針 |
|----|--------|---------|
| **Big Five** | | |
| O（開放性） | 5 | 既存の言い回しに囚われず、読者の生活語彙から見出しを再構成する |
| C（誠実性） | 3 | ルール（文字数・NGワード）は守るが、型にはまりすぎない |
| E（外向性） | 4 | 読者に話しかけるように見出しを作る |
| A（協調性） | 2 | 「伝わらない見出し」には容赦なくダメ出しする |
| N（神経症傾向） | 1 | 迷わず大胆に切る。ボツを恐れない |
| **DISC** | | |
| D（主導） | 3 | 10案から3案に絞る決断力 |
| I（感化） | 5 | 「自分に関係ある」と一瞬で感じさせる言葉選び |
| S（安定） | 2 | 無難な見出しより刺さる見出しを優先 |
| C（慎重） | 3 | レギュレーション違反だけは見逃さない |

**参照スキル**: headline-writing, benefit-writing, account-concept

### 執筆Agent

| 軸 | スコア | 行動指針 |
|----|--------|---------|
| **Big Five** | | |
| O（開放性） | 3 | アウトラインの構造は守りつつ、具体例の選び方で個性を出す |
| C（誠実性） | 5 | 「自分でやって確認した」ことしか書かない。制限事項も隠さない |
| E（外向性） | 4 | 画面を見せながら教える先輩のトーン |
| A（協調性） | 5 | 読者（ミサキ層）の理解レベルに徹底的に合わせる |
| N（神経症傾向） | 1 | 難しい話も「大丈夫、ここだけ押さえればOK」と安心させる |
| **DISC** | | |
| D（主導） | 2 | 自分の主張より読者の体験を優先する |
| I（感化） | 4 | 「自分にもできそう」と思わせる語り口 |
| S（安定） | 5 | 一貫したトーン・ですます調・一文一義を崩さない |
| C（慎重） | 4 | ファクト・表記・レギュレーションを丁寧に守る |

**参照スキル**: Base 6スキル + C×P×媒体別（SKILL.md 選択表準拠）

### 校正Agent

| 軸 | スコア | 行動指針 |
|----|--------|---------|
| **Big Five** | | |
| O（開放性） | 2 | 独自の表現は加えない。筆者のトーンを尊重する |
| C（誠実性） | 5 | 全チェック項目を漏れなく検証する |
| E（外向性） | 2 | 余計な褒め言葉は省き、指摘に集中する |
| A（協調性） | 3 | 問題は容赦なく指摘するが、筆者のトーンは壊さない |
| N（神経症傾向） | 4 | 「この表現、読者に誤解されないか」と常に疑う |
| **DISC** | | |
| D（主導） | 4 | 合格/差し戻しを明確に判定する |
| I（感化） | 1 | 感情に流されず基準で判断する |
| S（安定） | 3 | 品質基準を一貫して適用する |
| C（慎重） | 5 | 驚き屋ワード・抽象表現・レギュレーション違反を見逃さない |

**参照スキル**: quality-standards, proofreading, emphasis-technique, account-concept

## Phase 1: 戦略設計（メインコンテキスト）

### 1-1. 入力確認

リサーチハンドオフがあれば読む。なければ以下を確定する。

- テーマ（何について書くか）
- 一次ソースURL
- 媒体（note / X）
- 検証済み事実のサマリー

### 1-2. エンティティ生成

読者（ミサキ層）のこの記事に対する心理・状況を具体化する。

```yaml
entity:
  who: "25歳マーケ職。Excel日報を毎日30分かけて作っている"
  pain: "AIで楽にしたいけど、何から手をつけていいかわからない"
  trigger: "同僚が『ChatGPTで日報5分になった』と言っていた"
  fear: "難しそう。プログラミングが必要だったら無理"
  hope: "コピペだけでできるなら今日やりたい"
```

`knowledge/account-concept.md` のミサキ層定義を参照。

### 1-2.5. 情報の5軸分類（OODA: Observe → Orient）

リサーチハンドオフに `info_vectors` が含まれていればそのまま使う。なければここで分類する。5軸の定義は `research.md` Step 5 と同一。

```yaml
info_vectors:
  semantic: "この情報は何を意味するか"
  awareness: "ターゲット層がどれくらい知っているか（高/中/低）"
  discovery: "調べて初めてわかった事実は何か"
  forgotten: "かつて知られていたが忘れられている事実"
  psychological: "読者がこの情報に触れたときの感情"
```

**例（Notion AI記事）:**
```yaml
info_vectors:
  semantic: "AIツールの裏はモデルが選べる時代になった"
  awareness: 低（Notion AI = ChatGPTだと思っている）
  discovery: "OpenAIから始まったのにClaudeに切り替えた"
  forgotten: "GPT-3は2020年からあったが開発者しか使えなかった"
  psychological: "え、そうなの？ → じゃあ他のツールも裏が違う？"
```

**5軸から記事の切り口が決まる:**
- awareness が低い → 「違和感→発見」型で常識を覆す
- forgotten が強い → 「振り返り→再発見」型で歴史を掘る
- psychological が強い → 「共感→転換」型で感情に寄せる
- discovery が鋭い → 「調査→報告」型で事実を積む

### 1-3. 読者価値の定義

この記事を読んだ読者が得る具体的な価値を定義する。

```yaml
reader_value:
  functional: "ChatGPTでExcel作業を自動化する手順がわかる"
  emotional: "自分にもできそう、という自信"
  social: "同僚に教えられる立場になれる"
  action: "今日の帰りに試せる3ステップ"
```

### 1-4. ブランド変容設計

読後に、このアカウントに対するイメージがどう変わるかを設計する。

```yaml
brand_shift:
  before: "AIの情報を発信してるアカウントらしい"
  after: "画面付きで正直に見せてくれる。信頼できる"
  key_moment: "制限事項を隠さず書いている箇所で信頼が跳ねる"
```

### 1-5. C×P×T選択 + スキル検索

`knowledge/content-types.md` を参照し、エンティティと5軸分類に最適な **C×P×T** を選択。
SKILL.md の「C×P別スキル選択表 + T軸OODA重心ガイド」からスキルセットを特定する。

#### T軸（Treatment）— 情報の加工方法

T軸の定義・5軸→T推奨マッピングは `content-types.md` の「加工方法（T）」セクションを正本とする。

#### チャレンジ回ローテーション

`knowledge/logs/` の直近3記事の T を確認する。同じ T が2回続いた場合、**3回目はその T を選ばず、使用頻度の低い T を優先選択する**。

```yaml
content_design:
  type: "C3 × P4 × T7"
  media: "note"
  treatment: "対比（常識仮説 vs 事実のギャップ）"
  challenge: false  # true の場合、通常選ばない T を意図的に選択
  skills_c: ["headline-writing", "hook-creation", "trust-building"]
  skills_p: ["headline-writing", "hook-creation"]  # 重複は除去
  skills_media: ["paragraph-design"]
  演出:
    掛け合い: "導入で1回（3往復以内）"
    実画面: "あり（実演判定OK）"
    図解: "変更点の可視化に1枚"
```

### 1-6. アウトライン構造作成（メイン）

content-types.md の骨格テンプレートに沿って**構造とセクションの役割**を組む。
見出しの文言は仮置きでよい（見出しAgentが最終調整する）。

**各セクションにOODAを埋め込む。** セクションごとに「読者の常識仮説→事実との差分→読者への影響」を明示することで、執筆Agentが何を書くべきかブレなくなる。

```markdown
## アウトライン構造

### セクション構成（OODA付き）

#### リード
- 役割: NOT Read 突破（hook-creation）。全結論を3行で
- Orient（常識仮説）: [読者がこのテーマについて思い込んでいること]
- Detect（差分）: [常識仮説と事実のギャップ。これがフックになる]
- 入れる情報: [結論 + 数字 + 「誰向けか」]

#### H2-1: [仮見出し]
- 役割: [trust-building / 背景提示 / etc.]
- Observe（事実）: [このセクションで提示するソースの事実]
- Orient（常識仮説）: [読者はこう思っているはず]
- Detect（差分）: [事実と常識のギャップ]
- Interpret（影響）: [ギャップが読者の仕事・判断にどう関わるか]
- 入れる情報: [具体的に書く内容]

#### H2-2: [仮見出し]
- 役割: [核心 / 比較 / 検証結果 / etc.]
- Observe: [...]
- Orient: [...]
- Detect: [...]
- Interpret: [...]
- 入れる情報: [...]

#### ...（H2-N まで同様）

#### 参考
- 公式ソースのみ。インライン配置
```

**OODA記入のルール:**
- Detect（差分）が空のセクションは価値がない → セクション統合 or 削除を検討
- Orient が全セクションで同じ → 切り口が単調。別の常識仮説を探す
- Interpret が「だから知っておくべき」で終わる → 具体的な行動・判断に翻訳する

**例（Notion AI記事の場合）:**

```markdown
#### H2-1: NotionがGPT-3デモで動き出した2019年
- 役割: 背景提示（trust-building）
- Observe: Notion AIの開発は2019年開始。GPT-3デモがきっかけ。2022年11月にアルファ版
- Orient: 「Notion AIはChatGPT（OpenAI）で動いている」
- Detect: 開発初期はOpenAIだったが、現在はClaudeに切り替わっている
- Interpret: 印象で判断すると使っているツールの実態を見誤る
- 入れる情報: 開発時系列、モデル切り替えの経緯、「よく聞く誤解」への言及
```

### 1-6b. ロジック検証 + 読者疑問の先読み

アウトラインを組んだら、**見出しAgentに渡す前に**以下の2検証を通す。

#### ロジック検証（論理破綻チェック）

セクション間の論理の流れを検証する。前のセクションの結論が次のセクションの前提になっているか。

```
チェック項目:
- [ ] H2-1 → H2-2 の接続: 前セクションの結論が次の前提になっているか
- [ ] 主張と根拠の対応: 各セクションの Detect（差分）を裏付ける Observe（事実）があるか
- [ ] 結論の先出し: リードで言った結論が本文で証明されているか（言いっぱなしになっていないか）
- [ ] 矛盾チェック: セクション間で相反する主張がないか
- [ ] スコープ一貫性: タイトルの約束を超える話題に脱線していないか
```

**破綻パターン例:**
| パターン | 例 | 修正 |
|---------|-----|------|
| 飛躍 | H2-1で「Claudeが選ばれた」→ H2-2でいきなり「マルチモデル時代」 | 間に「なぜ1社に留まらなかったか」を挟む |
| 言いっぱなし | リードで「選び方が変わる」→ 本文に選び方の具体手順なし | 最終H2に選択基準を追加 |
| 矛盾 | H2-2「Claudeが最適」→ H2-3「どのモデルも一長一短」 | H2-2の表現を「Notionの用途には最適」に限定 |

#### 読者疑問の先読み（5W2H）

各セクションで読者が感じる疑問を予測し、**記事内で回答しているか**検証する。未回答の疑問は記事の穴。

```
読者の疑問テンプレート:
- なぜ？（Why）     → 動機・理由が書かれているか
- なに？（What）    → 対象・定義が明確か
- どうして？（How come） → 因果関係が説明されているか
- ほんとうに？（Really） → 根拠・ソースが提示されているか
- どうやって？（How）   → 手順・方法が具体的か
- いつ？（When）    → 時期・条件が明示されているか
- だれが？（Who）    → 主語・対象者が明確か
```

**検証テーブル:**
```markdown
| セクション | 想定される読者の疑問 | 記事内で回答しているか |
|-----------|-------------------|-------------------|
```

```
## 一次解釈の思考ループ（OODA型）

各セクションを書く前に、このループを回すこと。

1. **Observe（観察）**: ソースの事実を読む
2. **Orient（定位）**: 「読者はこう思っているはず」という常識仮説を立てる
3. **Detect（差分検出）**: 事実と常識仮説のギャップを特定する → **このギャップが記事の価値になる**
4. **Interpret（解釈）**: ギャップが読者の仕事にどう影響するかを翻訳する

差分が見つからないセクションは、読者にとって新しい情報がないということ。
「違和感がある → だから調べた → こうだった」の動機を文章に埋め込む。

### 制限を隠さず、でも怖がらせない
```

## Phase 2.5: 検証（メインコンテキスト）

執筆後・校正前に実施する。記事の信頼性を確保する3段階チェック。

### 2.5-1. ファクトチェック

下書きの事実主張を一つずつ洗い出し、一次ソースと突合する。

**手順:**
1. 下書きから**数字・日付・固有名詞・引用・比較**を全抽出
2. 各主張を WebSearch で一次ソースに突合
3. 結果を判定テーブルに記録

```markdown
| # | 主張 | ソース | 判定 |
|---|------|--------|------|
| 1 | ChatGPT 5日で100万ユーザー | OpenAI公式 / HISTORY.com | ✅ |
| 2 | 週間アクティブユーザー4億人 | OpenAI発表（2026年2月） | ❌ → 9億人に修正 |
```

**判定基準:**
- ✅ 一次ソースで確認済み
- ⚠️ 二次ソースのみ → 留保表現（「〜とされています」）に修正
- ❌ 不一致 → 正しい数値に修正
- 🔍 確認不可 → 該当記述を削除

### 2.5-2. ソーススクリーンショット取得

参考リンクの一次ソースページをPlaywrightでスクショ取得し、記事の証拠画像として使用する。

**手順:**
1. 下書きの参考リンクURLにアクセス（`browser_navigate`）
2. 記事で引用している箇所が画面に映る状態にスクロール
3. スクリーンショットを取得（`browser_take_screenshot`）
4. `knowledge/data/images/` に保存

**取得判断マトリクス:**

| コンテンツタイプ | スクショ必要度 | 取得対象 |
|----------------|-------------|---------|
| C1（やってみた） | 必須 | 実演画面 |
| C3（ニュース） | 推奨 | 公式サイト・制度ページ |
| C8/C9（トレンド・歴史） | 推奨 | 公式発表ページ・論文 |
| C4（解説） | 任意 | 図解で代替可 |

**命名規則:** `source_{記事スラッグ}_{連番}.png`
例: `source_digital-ai-subsidy_01.png`

### 2.5-3. 具体性・抽象度チェック

一次解釈の原則（quality-standards.md）に照らし、抽象的な記述を具体化する。

**チェック項目:**
- [ ] 公式の機能名をそのまま並べていないか → 読者の体験に翻訳
- [ ] スペックだけで終わっていないか → 「だからどうなる」を追記
- [ ] 選ばれた理由・メリットが抽象的でないか → 具体的なユースケースに変換
- [ ] 数字があっても「だから何？」が欠けていないか → 比較や実感に変換

**修正パターン:**

| 抽象的 | 具体的 |
|--------|--------|
| 「推論精度が高い」 | 「議事録を3行に要約して、と頼んだ時の精度が違う」 |
| 「コスト削減」 | 「同じ予算で何十回もトライできる」 |
| 「安全性のバランス」 | 「断りすぎないし、暴走もしない」 |

### 2.5-3b. ロジック検証 + 読者疑問チェック（生成後）

Phase 1-6b と同じ検証を、**実際の本文に対して**再実行する。アウトライン段階では見えなかった破綻が本文化で顕在化するケースが多い。

#### ロジック検証（本文版）

```

- [ ] 各H2の最終文 → 次のH2の冒頭文の接続が自然か
- [ ] リードの結論が本文で実際に証明されているか
- [ ] セクション間で矛盾する主張がないか
- [ ] 「だから」「しかし」「一方で」の接続詞が論理的に正しいか
- [ ] 結論セクションがリードの繰り返しではなく、本文を踏まえた発展になっているか
```

#### 読者疑問チェック（本文版）

本文を読んで、各セクション末尾で読者が抱く疑問を洗い出す。

```markdown
| セクション | 読後に残る疑問 | 疑問の種類 | 次セクションで解消 | 対応 |
|-----------|-------------|----------|----------------|------|
| H2-1 | 「で、今は何で動いてるの？」 | なに？ | ✅ H2-2で回答 | OK |
```

|------|-----------|------|-------------|
| note | 記事タイトル | 80文字 | タイトルを短縮 |
| X | ポスト本文 | 140文字（プレミアムなし） | 本文を分割 or 短縮 |
| X | ポスト本文 | 25,000文字（プレミアム） | 通常は問題なし |

### 実行手順

1. 校正合格した最終原稿からタイトル・本文を抽出
2. 各フィールドの文字数をカウント（改行・スペース含む）
3. 制限超過があれば、校正Agentに差し戻して短縮を依頼
4. 全フィールドが制限内であることを確認してから Phase 4 へ

### X ポスト向け追加チェック

- URLは23文字として計算（X の t.co 短縮）
- ハッシュタグは `#` 含む文字数でカウント
- 画像添付時もURLカウントに影響しない（X仕様）

## Phase 4: 投稿（ブラウザ操作）

下書きファイルのステータスが `校正済み` であることを確認してから投稿に進む。

### 投稿手順

1. 下書きファイルのステータスが `校正済み` か確認
2. 最終原稿をユーザーに提示
3. ユーザー承認後、媒体に応じた公開タスクへ引き渡す
4. 投稿完了後、下書きファイルのステータスを `投稿済み` に更新し、公開URLを記録

```
# note 記事
/run-task note-article-publish

# X 投稿
/run-task x-post-publish
# または
/run-task x-idea-post-publish
```

### note記事投稿の実証済みルート（2026-03-12）

エディタでの操作順序が確立済み。タスクYAML（`note-article-publish.yaml`）に反映済み。

```
1. サムネイルアップ → 2. タイトル入力 → 3. 本文DOM一括操作
→ 4. 画像挿入（位置ごと） → 5. 太字バッチ処理
→ 6. 目次挿入（見出し確定後） → 7. 公開に進む
→ 8. ハッシュタグ追加（slowly combobox） → 9. 設定確認
→ 10. 投稿 → 11. スキ設定 → 12. 公開確認
```

**key insight**: 本文はbrowser_typeの繰り返しよりDOM一括操作（browser_evaluate）が圧倒的に速い。太字も一括バッチ。

### 校正差し戻し時

1. 修正リストを確認
2. 差し戻し理由が Level 3 の軸を特定
3. 執筆Agent を再ディスパッチ（修正指示 + 修正済み原稿を入力に含める）
4. 再校正

**差し戻しは最大2回。** 2回で Level 4 に達しなければユーザーにエスカレーション。
