# /generate-content ワークフロー

リサーチハンドオフ（または直接テーマ指示）を受け取り、**下書きファイルを中心に磨き上げてから**投稿する。

**プラットフォーム別ワークフロー:**

| 媒体 | 検証・公開 | ショートカット |
|------|-----------|-------------|
| note | → `generate-content-note.md`（Phase 2.5〜4） | — |
| X | → `x-post.md`（テーマ→投稿を一気通貫） | `/x-post` |

note記事は本ワークフローの Phase 1→2→(note固有 2.5→3→3.5→4) で進む。
X投稿は `/x-post` で独立して完結する。本ワークフロー経由でも可能だが非推奨。

---

## 下書きファースト原則

記事は必ず `knowledge/drafts/` にMarkdownファイルとして作成し、磨き上げてからnote/Xに投稿する。
**下書きファイルが全工程の中心成果物。** ブラウザ投稿は最後の転記作業にすぎない。

```
knowledge/drafts/
├── note_{スラッグ}.md      ← note記事の下書き
├── x_{スラッグ}.md         ← X投稿の下書き
└── ...
```

### 下書きファイルの構造

```markdown
# 下書き: {タイトル}

- **C×P×T**: {C×P×T}
- **ステータス**: 下書き | 検証済み | 校正済み | 投稿済み
- **想定文字数**: {N}〜{N}文字
- **ハッシュタグ**: #{tag1} #{tag2}
- **タイトル文字数**: {N}文字

## 画像

| # | 種別 | ファイル | 挿入位置 | 検証 |
|---|------|---------|---------|------|
| 1 | サムネイル | `knowledge/data/images/note_thumbnail_{slug}_{YYYY-MM-DD}.jpg` | 記事トップ | [ ] |

---

{リード}

---

{本文}

---

※ 参考リンクは末尾にまとめず、関連セクションの直下にインライン配置する
```

### ステータス遷移

```
下書き → 検証済み → 校正済み → 投稿済み
  ↑         |          |
  └─ 差し戻し ←─────────┘
```

| ステータス | 意味 | 次のアクション |
|-----------|------|---------------|
| 下書き | Phase 2 完了。本文あり | Phase 2.5（検証）へ |
| 検証済み | ファクトチェック・スクショ・具体性OK | Phase 3（校正）へ |
| 校正済み | 品質 Level 4+ 確認済み | ユーザー承認 → Phase 4（投稿）へ |
| 投稿済み | noteまたはXに公開完了 | URL記録・ログ更新 |

### セッション分割の標準パターン

記事の作成と投稿は**別セッションでもよい**。下書きファイルが状態を保持するので、途中で中断しても次のセッションで再開できる。

```
セッション A: テーマ指示 → Phase 1〜2 → 下書き作成
セッション B: 下書き Read → Phase 2.5〜3 → 磨き上げ
セッション C: 校正済み下書き → Phase 4 → note投稿
```

**複数記事のバッチ作成**: 下書きを先に全部作り、まとめて検証・校正するのも有効。ただし執筆は1記事ずつ順番に行う（並列執筆は品質が崩れる）。

## 使い方

```
/generate-content <テーマ or ハンドオフ> [媒体: note|x] [C×P×T指定]
/generate-content --resume <draft_path>              ← 既存下書きから再開
/generate-content --resume <draft_path> --phase 2.5  ← 指定フェーズから再開
```

例:
- `/generate-content "ChatGPT for Excel" 媒体: note C3×P4`
- `/generate-content handoff: knowledge/logs/research/2026-03-12.md#candidate-1`
- `/generate-content --resume knowledge/drafts/note_chatgpt-history.md`
- `/generate-content --resume knowledge/drafts/note_chatgpt-history.md --phase 3`

### --resume モード

既存の下書きファイルから途中再開する。下書きのステータスから開始 Phase を自動判定する。

| 下書きステータス | 自動開始 Phase | やること |
|----------------|---------------|---------|
| 下書き | Phase 2.5 | 検証から |
| 検証済み | Phase 3 | 校正から |
| 校正済み | Phase 4 | 投稿へ |

`--phase` で明示指定すると自動判定を上書きする。Phase 1 を指定するとアウトラインから作り直す（下書きは上書きされる）。

| Agent | プロンプト | 性格の要点 |
|-------|----------|-----------|

## Phase 1: 戦略設計（メインコンテキスト）

### 1-1. 入力確認

リサーチハンドオフがあれば読む。なければ以下を確定する。

- テーマ（何について書くか）
- 一次ソースURL
- 媒体（note / X）
- 検証済み事実のサマリー

### 1-2. エンティティ生成

読者（ミサキ層）のこの記事に対する心理・状況を具体化する。

```yaml
entity:
  who: "25歳マーケ職。Excel日報を毎日30分かけて作っている"
  pain: "AIで楽にしたいけど、何から手をつけていいかわからない"
  trigger: "同僚が『ChatGPTで日報5分になった』と言っていた"
  fear: "難しそう。プログラミングが必要だったら無理"
  hope: "コピペだけでできるなら今日やりたい"
```

`knowledge/account-concept.md` のミサキ層定義を参照。

### 1-2.5. 情報の5軸分類（OODA: Observe → Orient）

リサーチハンドオフに `info_vectors` が含まれていればそのまま使う。なければここで分類する。5軸の定義は `research.md` Step 5 と同一。

```yaml
info_vectors:
  semantic: "この情報は何を意味するか"
  awareness: "ターゲット層がどれくらい知っているか（高/中/低）"
  discovery: "調べて初めてわかった事実は何か"
  forgotten: "かつて知られていたが忘れられている事実"
  psychological: "読者がこの情報に触れたときの感情"
```

**5軸から記事の切り口が決まる:**
- awareness が低い → 「違和感→発見」型で常識を覆す
- forgotten が強い → 「振り返り→再発見」型で歴史を掘る
- psychological が強い → 「共感→転換」型で感情に寄せる
- discovery が鋭い → 「調査→報告」型で事実を積む

### 1-3. 読者価値の定義

```yaml
reader_value:
  functional: "ChatGPTでExcel作業を自動化する手順がわかる"
  emotional: "自分にもできそう、という自信"
  social: "同僚に教えられる立場になれる"
  action: "今日の帰りに試せる3ステップ"
```

### 1-4. ブランド変容設計

```yaml
brand_shift:
  before: "AIの情報を発信してるアカウントらしい"
  after: "画面付きで正直に見せてくれる。信頼できる"
  key_moment: "制限事項を隠さず書いている箇所で信頼が跳ねる"
```

### 1-5. C×P×T選択 + スキル検索

`knowledge/content-types.md` を参照し、エンティティと5軸分類に最適な **C×P×T** を選択。
SKILL.md の「C×P別スキル選択表 + T軸OODA重心ガイド」からスキルセットを特定する。

#### チャレンジ回ローテーション

`knowledge/logs/` の直近3記事の T を確認する。同じ T が2回続いた場合、**3回目はその T を選ばず、使用頻度の低い T を優先選択する**。

```yaml
content_design:
  type: "C3 × P4 × T7"
  media: "note"
  treatment: "対比（常識仮説 vs 事実のギャップ）"
  challenge: false  # true の場合、通常選ばない T を意図的に選択
  skills_c: ["headline-writing", "hook-creation", "trust-building"]
  skills_p: ["headline-writing", "hook-creation"]
  skills_media: ["paragraph-design"]
  演出:
    掛け合い: "導入で1回（3往復以内）"
    実画面: "あり（実演判定OK）"
    図解: "変更点の可視化に1枚"
```

### 1-6. アウトライン構造作成（メイン）

content-types.md の骨格テンプレートに沿って**構造とセクションの役割**を組む。
見出しの文言は仮置きでよい（見出しAgentが最終調整する）。

**各セクションにOODAを埋め込む。** セクションごとに「読者の常識仮説→事実との差分→読者への影響」を明示することで、執筆Agentが何を書くべきかブレなくなる。

```markdown
## アウトライン構造

### セクション構成（OODA付き）

#### リード
- 役割: NOT Read 突破（hook-creation）。全結論を3行で
- Orient（常識仮説）: [読者がこのテーマについて思い込んでいること]
- Detect（差分）: [常識仮説と事実のギャップ。これがフックになる]
- 入れる情報: [結論 + 数字 + 「誰向けか」]

#### H2-1: [仮見出し]
- 役割: [trust-building / 背景提示 / etc.]
- Observe（事実）: [このセクションで提示するソースの事実]
- Orient（常識仮説）: [読者はこう思っているはず]
- Detect（差分）: [事実と常識のギャップ]
- Interpret（影響）: [ギャップが読者の仕事・判断にどう関わるか]
- 入れる情報: [具体的に書く内容]

#### ...（H2-N まで同様）

#### 参考
- 公式ソースのみ。インライン配置
```

**OODA記入のルール:**
- Detect（差分）が空のセクションは価値がない → セクション統合 or 削除を検討
- Orient が全セクションで同じ → 切り口が単調。別の常識仮説を探す
- Interpret が「だから知っておくべき」で終わる → 具体的な行動・判断に翻訳する

### 1-6b. ロジック検証 + 読者疑問の先読み

アウトラインを組んだら、**見出しAgentに渡す前に**以下の2検証を通す。

#### ロジック検証（論理破綻チェック）

```
チェック項目:
- [ ] H2-1 → H2-2 の接続: 前セクションの結論が次の前提になっているか
- [ ] 主張と根拠の対応: 各セクションの Detect（差分）を裏付ける Observe（事実）があるか
- [ ] 結論の先出し: リードで言った結論が本文で証明されているか
- [ ] 矛盾チェック: セクション間で相反する主張がないか
- [ ] スコープ一貫性: タイトルの約束を超える話題に脱線していないか
```

#### 読者疑問の先読み（5W2H）

各セクションで読者が感じる疑問を予測し、**記事内で回答しているか**検証する。

```markdown
| セクション | 想定される読者の疑問 | 記事内で回答しているか |
|-----------|-------------------|-------------------|
| リード | 「本当にChatGPTじゃないの？」（ほんとうに？） | ✅ ソース明記 |
| H2-1 | 「なんで切り替えたの？」（なぜ？） | ✅ 比較評価の経緯 |
| H2-2 | 「具体的に何が違うの？」（なに？） | ⚠️ 抽象的 → 具体例追加 |
```

### 1-7. 見出し作成

タイトルとH2見出しの作成。**推奨は見出し+執筆の統合モード**（Phase 2参照）。

#### 見出しルール

- タイトル: 28〜32文字。10案以上出し、ベスト1案を自分で選ぶ（ユーザーに選ばせない）
- 【】フックはタイトルか最初のH2の1箇所のみ。全見出しに入れない
- H2: 情報が一目でわかる素直な一文。ダッシュ（—）は使わない
- 制限・ネガティブ情報は読者の欲求に変換する
- 驚き屋ワード禁止（やばい、神、革命、衝撃）
- ミサキ層が「自分に関係ある」と思える言葉を使う

#### 見出しAgentプロンプト（分離実行時のみ使用）

→ `.claude/workflows/prompts/headline-agent.md` を Read し、変数を展開してサブエージェントに渡す。

### 1-8. レギュレーション確認

`quality-standards.md` のチェックリストでアウトラインを事前検証。

- [ ] 目的の階層（L1〜L5）が定義されているか
- [ ] エンティティに合った語彙レベルか
- [ ] 驚き屋ワードが入っていないか
- [ ] 結論がリードに入っているか
- [ ] 参考リンクは公式ソースのみか

### ユーザーフィードバック統合チェックリスト

以下は過去のセッションで繰り返し指摘されたルール。**全フェーズで準拠必須**。

| # | ルール | チェックタイミング |
|---|--------|------------------|
| F1 | **前置き禁止**: 「正直に書きます」等のメタコメンタリーを入れない | Phase 2（執筆） |
| F2 | **説明ベース+画像で崩す**: 本文は落ち着いた情報提供、画像・図解で視覚的に崩す | Phase 1-6（アウトライン） |
| F3 | **Level 3 = 不合格**: Qiita/Zenn品質では存在意義がない。Level 4以上必須 | Phase 3（校正） |
| F4 | **【】フックは1箇所のみ**: タイトルか最初のH2だけ | Phase 1-7（見出し） |
| F5 | **見出しにダッシュ（—）禁止** | Phase 1-7（見出し） |
| F6 | **制限→欲求変換**: ネガティブ情報は読者の欲求に変換 | Phase 1-7（見出し） |
| F7 | **一般化した悩みの描写不要**: 読者が知っている問題の丁寧な描写に文字数を使わない | Phase 2（執筆） |
| F8 | **一次解釈必須**: 公式情報をそのまま並べず、使っている人の目線で解釈 | Phase 2（執筆） |

| F10 | **見出し・タイトルにキャラ名禁止** | Phase 1-7（見出し） |
| F11 | **既知の常識を投稿しない**: 難しい話を噛み砕く方向が正 | Phase 1-1 |
| F12 | **参考リンクはインライン配置**: 末尾に「## 参考」まとめNG | Phase 2（執筆） |
| F13 | **サムネイルにテキスト必須**: 装飾的イラストNG | Phase 2.5 |
| F14 | **記事執筆は1つずつ**: 並列執筆は品質が崩れる | 全体 |
| F15 | **選択肢を聞かない**: AI側で判断して進める | Phase 1-7 |
| F16 | **サブエージェントには人格を与える**: Big Five × DISC | Phase 2, 3 |
| F17 | **記事執筆フロー厳守**: リサーチ→スキル→エンティティ→C×P→演出→執筆 | Phase 1 全体 |

### 1-9. ユーザー承認ゲート

アウトライン + エンティティ + 価値定義 + ブランド変容をユーザーに提示。
**承認を得てから Phase 2 に進む。**

#### スキップ条件

| 条件 | 例 |
|------|---|
| **バッチモード** | ユーザーが複数記事の連続生成を指示 |
| **明示的スキップ指示** | 「進めて」「そのまま書いて」 |
| **同セッション内の再生成** | 方向性は既に合意済み |

## Phase 2: 執筆（サブエージェント）

### 出力先

| 媒体 | 下書きパス |
|------|-----------|
| note | `knowledge/drafts/note_{スラッグ}.md` |
| x    | `knowledge/drafts/x_{スラッグ}.md` |

### 推奨: 見出し+執筆 統合モード

見出しAgentと執筆Agentを**1つのサブエージェント呼び出しに統合する**のが推奨パターン。

**統合する理由:**
- スキルファイルの再読み込みが不要（コンテキスト効率）
- 見出しの意図が執筆に直接引き継がれる（情報ロスなし）
- 1回の Agent 呼び出しで完結する

**分離する場合（例外）:**
- 校正で差し戻された → 見出しは維持、本文だけ再生成
- 見出しだけ先に承認を取りたい場合

### 執筆Agentプロンプト

→ `.claude/workflows/prompts/writing-agent.md` を Read し、変数を展開してサブエージェントに渡す。

### 変数解決（見出し・執筆共通）

| 変数 | 解決方法 |
|------|---------|
| `{SKILL_DIR}` | `.agent/skills/god-writing` の絶対パス |
| `{KNOWLEDGE_DIR}` | `knowledge` の絶対パス |
| `{RESOLVED_SKILLS}` | Phase 1-5 で特定したスキルファイルの絶対パスリスト |
| `{ENTITY_YAML}` | Phase 1-2 の出力 |
| `{READER_VALUE_YAML}` | Phase 1-3 の出力 |
| `{BRAND_SHIFT_YAML}` | Phase 1-4 の出力 |
| `{OUTLINE}` / `{OUTLINE_STRUCTURE}` | Phase 1-6 の出力 |
| `{VERIFIED_FACTS}` | リサーチログの検証済み事実リスト |
| `{CORRECTIONS}` | リサーチログの修正事項。新規記事では空 |

## Phase 3: 校正（サブエージェント）— 共通

校正は note / X 共通のプロセス。ただし note は Phase 2.5 検証後、X は Phase 2 直後に実行。

### 校正Agentプロンプト

→ `.claude/workflows/prompts/proofreading-agent.md` を Read し、変数を展開してサブエージェントに渡す。

| 変数 | 解決方法 |
|------|---------|
| `{SKILL_DIR}` | `.agent/skills/god-writing` の絶対パス |
| `{KNOWLEDGE_DIR}` | `knowledge` の絶対パス |
| `{DRAFT_TEXT}` | Phase 2 の出力（下書き本文） |
| `{ENTITY_YAML}` | Phase 1-2 の出力 |
| `{BRAND_SHIFT_YAML}` | Phase 1-4 の出力 |

### 差し戻し

1. 修正リストを確認
2. 差し戻し理由が Level 3 の軸を特定
3. 執筆Agent を再ディスパッチ（修正指示 + 修正済み原稿を入力に含める）
4. 再校正

**差し戻しは最大2回。** 2回で Level 4 に達しなければユーザーにエスカレーション。
